---
title: 버전 관리 정책
---

이 페이지는 `react-native` 패키지에 적용되는 버전 관리 정책을 설명합니다.

React Native의 각 버전은 수동 및 자동화된 테스트를 통해 철저히 검증하여 품질이 저하되지 않도록 합니다.

React Native의 `stable` 채널은 아래에 설명된 0.x.y 릴리즈 정책을 따릅니다.

React Native는 실험적인 기능에 대한 조기 피드백을 장려하기 위해 `nightly` 릴리즈 채널도 제공합니다.

이 페이지는 `react-native` 및 `@react-native` 스코프 하위 패키지들의 버전 번호 방식을 설명합니다.

## 안정 릴리즈 버전

React Native는 정기적인 주기로 안정 버전을 릴리즈합니다.

0.x.y 버전 관리 체계를 따릅니다:

- 브레이킹 체인지는 새로운 마이너 버전으로 출시됩니다. 즉, x 숫자를 증가시킵니다(예: 0.78.0에서 0.79.0).
- 새로운 기능 및 API도 새로운 마이너 버전으로 출시됩니다. 즉, x 숫자를 증가시킵니다(예: 0.78.0에서 0.79.0).
- 치명적인 버그 수정은 새로운 패치 버전으로 출시됩니다. 즉, y 숫자를 증가시킵니다(예: 0.78.1에서 0.78.2).

안정 릴리즈는 정기적으로 출시되며, 최신 버전은 NPM에서 `latest`로 태그됩니다.

동일한 마이너 번호 하위의 릴리즈 시리즈를 **마이너 시리즈**라고 합니다(예: 0.76.x는 0.76.0, 0.76.1, 0.76.2 등의 마이너 시리즈입니다).

**안정성에 대한 약속**에 대해서는 [릴리즈 페이지](./)에서 자세히 확인할 수 있습니다.

### 브레이킹 체인지

브레이킹 체인지는 모든 사람에게 불편하며, 최소화하기 위해 노력하고 있습니다. 각 안정 릴리즈에 포함되는 모든 브레이킹 체인지는 다음에서 강조됩니다:

- [React Native Changelog](https://github.com/facebook/react-native/blob/main/CHANGELOG.md)의 _Breaking_ 및 _Removed_ 섹션
- 각 릴리즈 블로그 포스트의 _Breaking Changes_ 섹션

각 브레이킹 체인지에 대해 변경 이유를 설명하고, 가능하면 대체 API를 제공하며, 최종 사용자에게 미치는 영향을 최소화하는 것에 최선을 다합니다.

### 브레이킹 체인지란 무엇인가요?

React Native에서 브레이킹 체인지로 간주하는 것:

- 호환되지 않는 API 변경(즉, 해당 변경으로 인해 코드가 더 이상 컴파일/실행되지 않도록 변경되거나 제거된 API). 예시:
  - 코드가 컴파일되려면 변경이 필요한 JS/Java/Kotlin/Obj-c/C++ API의 변경.
  - 이전 버전과 호환되지 않는 `@react-native/codegen` 내부 변경.
- 중요한 동작/런타임 변경. 예시:
  - props의 레이아웃 로직이 대폭 변경되는 경우.
- 개발 경험의 중요한 변경. 예시:
  - 디버깅 기능이 완전히 제거되는 경우.
- 전이적 의존성의 주요 버전 업. 예시:
  - React를 18.x에서 19.x로 업그레이드
  - Android Target SDK를 34에서 35로 업그레이드.
- 지원 플랫폼 버전의 축소. 예시:
  - Android 최소 SDK를 21에서 23으로 올리는 경우
  - 최소 iOS 버전을 15.1로 올리는 경우.

브레이킹 체인지로 간주하지 않는 것:

- `unstable_` 접두사로 시작하는 API 수정: 이러한 API는 실험적인 기능을 노출하며, 최종 형태에 대한 확신이 없습니다. `unstable_` 접두사와 함께 릴리즈함으로써 더 빠르게 반복하고 안정적인 API에 빠르게 도달할 수 있습니다.
- 비공개 또는 내부 API의 변경: 이러한 API는 종종 `internal_`, `private_` 접두사가 붙거나 `internal/` 또는 `private/` 폴더/패키지 내에 있습니다. 툴링 제약으로 인해 일부 API가 공개 가시성을 가질 수 있지만, 이를 공개 API의 일부로 간주하지 않으므로 사전 공지 없이 변경합니다.
  - 마찬가지로, `__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED` 또는 `__reactInternalInstance$uk43rzhitjg`와 같은 내부 속성명에 접근하는 경우 어떠한 보장도 없습니다. 본인 책임 하에 사용하세요.
  - `@FrameworkAPI`로 어노테이션된 클래스도 내부로 간주됩니다.
- 툴링/개발 API의 변경: React Native의 일부 공개 API는 프레임워크 및 기타 도구와의 통합을 위해 예약되어 있습니다. 예를 들어, 일부 Metro API 또는 React Native DevTools API는 다른 프레임워크나 도구에서만 사용하도록 설계되어 있습니다. 이러한 API의 변경은 영향을 받는 도구들과 직접 협의하여 진행되며, 브레이킹 체인지로 간주되지 않습니다(릴리즈 블로그 포스트에서 폭넓게 공지하지 않습니다).
- 개발 경고: 경고는 런타임 동작에 영향을 주지 않으므로, 버전 간에 새로운 경고를 추가하거나 기존 경고를 수정할 수 있습니다.

변경이 커뮤니티에 광범위한 문제를 일으킬 것으로 예상되는 경우, 에코시스템에 점진적인 마이그레이션 경로를 제공하기 위해 최선을 다합니다.

### 사용 중단(Deprecation) 주기

React Native를 계속 개발하고 발전시키면서 새로운 API를 작성하고 기존 API를 사용 중단해야 하는 경우가 있습니다. 해당 API는 사용 중단 주기를 거치게 됩니다.

API가 사용 중단되면, **이후** 안정 릴리즈에서도 계속 사용할 수 있습니다.

예시: API가 React Native 0.76.x에서 사용 중단되면, 0.77.x에서도 여전히 사용할 수 있으며 React Native 0.78.x 이전에는 제거되지 않습니다.

에코시스템이 마이그레이션하기 위한 시간이 더 필요하다고 판단될 경우, 사용 중단된 API를 더 오랫동안 유지하기도 합니다. 이러한 API에 대해서는 일반적으로 사용자가 마이그레이션할 수 있도록 경고를 제공합니다.

## 릴리즈 채널

React Native는 번성하는 오픈소스 커뮤니티에 의존하여 버그 리포트, pull request, RFC를 제출받습니다. 피드백을 장려하기 위해 여러 릴리즈 채널을 지원합니다.

:::note
이 섹션은 프레임워크, 라이브러리 또는 개발자 툴링을 개발하는 개발자에게 가장 관련이 있습니다. 주로 사용자 대상 애플리케이션을 빌드하기 위해 React Native를 사용하는 개발자는 latest 이외의 릴리즈 채널에 대해 걱정할 필요가 없습니다.
:::

### latest

`latest`는 안정적인 semver React Native 릴리즈를 위한 채널입니다. npm에서 React Native를 설치할 때 받는 버전입니다. 지금 이미 사용하고 있는 채널입니다. React Native를 직접 사용하는 사용자 대상 애플리케이션은 이 채널을 사용합니다.

정기적으로 React Native의 새로운 마이너 시리즈를 게시하며, `latest` 태그를 최신 안정 버전으로 업데이트합니다.

### next

새로운 React Native 릴리즈를 안정 버전으로 선언하기 전에, RC0부터 시작하는 일련의 **릴리즈 후보(release candidate)**를 게시합니다. 이러한 버전은 사전 릴리즈 버전(`0.79.0-rc.0` 버전 관리 체계를 따름)이며 NPM에서 `next`로 태그됩니다.

새 브랜치가 분기되고 RC가 NPM 및 GitHub에 게시되기 시작하면, 라이브러리/프레임워크를 React Native의 `next` 버전에서 테스트해보는 것이 좋습니다.

이를 통해 프로젝트가 다가오는 React Native 버전에서도 계속 잘 작동할 수 있습니다.

그러나 사전 릴리즈/RC는 프로덕션 준비가 완료된 것으로 간주되지 않으므로 사용자 대상 애플리케이션에서 직접 사용하지 마세요.

### nightly

`nightly` 릴리즈 채널도 제공합니다. Nightly는 [facebook/react-native](https://github.com/facebook/react-native)의 `main` 브랜치에서 매일 게시됩니다. Nightly는 불안정한 React Native 버전으로 간주되며 프로덕션 사용에 권장하지 않습니다.

Nightly는 `0.80.0-nightly-<DATE>-<SHA>` 버전 관리 체계를 따르며, `<DATE>`는 nightly의 날짜이고 `<SHA>`는 해당 nightly 게시에 사용된 커밋의 SHA입니다.

Nightly 릴리즈는 테스트 목적으로만 제공되며, nightly 간에 동작이 변경되지 않는다는 보장을 제공하지 않습니다. latest/next에서의 릴리즈에 사용하는 semver 프로토콜을 따르지 않습니다.

매일 `react-native@nightly` 버전에서 라이브러리를 테스트하는 CI 워크플로우를 설정하면, 라이브러리가 향후 릴리즈에서도 계속 잘 작동하는지 확인하는 데 도움이 됩니다.
