풀 리퀘스트를 검토하는 데는 상당한 시간이 걸릴 수 있습니다. 경우에 따라 검토에 걸리는 시간이 누군가가 변경 사항을 작성하고 제출하는 데 걸린 시간보다 더 길 수도 있습니다! 따라서 각 풀 리퀘스트가 검토될 준비가 된 좋은 상태인지 확인하기 위해 사전 작업이 필요합니다.

풀 리퀘스트는 세 가지 주요 섹션으로 구성되어야 합니다:

- 요약. 이는 변경 사항의 동기를 이해하는 데 도움이 됩니다.
- Changelog. 이는 릴리즈 노트 작성에 도움이 됩니다. 또한 변경 사항의 간단한 요약 역할도 합니다.
- 테스트 계획. 이것이 풀 리퀘스트에서 가장 중요한 부분일 수 있습니다. 테스트 계획은 검토자가 변경 사항이 의도한 대로 작동하는지 검증할 수 있도록 재현 가능한 단계별 가이드여야 합니다. 사용자에게 보이는 변경 사항의 경우 스크린샷이나 동영상을 첨부하는 것도 좋습니다.

모든 풀 리퀘스트는 여러분이 익숙하지 않을 수 있는 React Native의 일부 영역에 대한 심층적인 이해를 필요로 할 수 있습니다. 풀 리퀘스트를 검토하기에 적합한 사람이 아니라고 느껴지더라도, 레이블을 추가하거나 작성자에게 더 많은 정보를 요청하는 방식으로 도움을 줄 수 있습니다.

## PR 검토하기

풀 리퀘스트는 병합되기 전에 GitHub의 검토 기능을 사용하여 검토 및 승인되어야 합니다. 누구나 풀 리퀘스트를 검토하고 승인할 수 있지만, 일반적으로 [기여자](https://github.com/facebook/react-native/blob/main/ECOSYSTEM.md) 중 한 명으로부터 승인이 올 때 풀 리퀘스트가 병합될 준비가 된 것으로 간주합니다.

자신 있게 검토할 수 있는 풀 리퀘스트를 찾았다면, GitHub의 검토 기능을 활용하여 명확하고 정중하게 제안하는 변경 사항을 전달하세요.

changelog나 테스트 계획이 부족하다고 표시된 풀 리퀘스트부터 시작하는 것을 고려해 보세요.

- [changelog가 없는 것으로 보이는 PR](https://github.com/facebook/react-native/pulls?utf8=%E2%9C%93&q=is%3Apr+is%3Aopen+label%3A%22Missing+Changelog%22+) - 살펴보고 PR을 편집하여 직접 changelog를 추가할 수 있는지 확인하세요. 그런 다음 "Missing Changelog" 레이블을 제거하세요.
- [테스트 계획이 없는 PR](https://github.com/facebook/react-native/pulls?q=is%3Apr+label%3A%22Missing+Test+Plan%22+is%3Aclosed) - 풀 리퀘스트를 열고 테스트 계획을 찾아보세요. 테스트 계획이 충분해 보이면 "Missing Test Plan" 레이블을 제거하세요. 테스트 계획이 없거나 불완전해 보이면 작성자에게 정중하게 테스트 계획 추가를 고려해 달라는 댓글을 남기세요.

풀 리퀘스트는 병합되기 전에 모든 테스트를 통과해야 합니다. 테스트는 `main`의 모든 커밋과 풀 리퀘스트에서 실행됩니다. 풀 리퀘스트가 검토 준비가 되도록 돕는 빠른 방법은 [pre-commit 테스트에 실패한 풀 리퀘스트를 검색](https://github.com/facebook/react-native/pulls?utf8=%E2%9C%93&q=is%3Apr+is%3Aopen+label%3A%22CLA+Signed%22+status%3Afailure+)하고 수정이 필요한지 여부를 결정하는 것입니다. 실패한 테스트는 일반적으로 스레드 하단 "Some checks were not successful." 아래에 나열됩니다.

- [main의 최신 테스트 실행 결과](https://circleci.com/gh/facebook/react-native/tree/main)를 빠르게 살펴보세요. `main`이 초록색인가요? 그렇다면:
  - 실패가 이 풀 리퀘스트의 변경 사항과 관련이 있어 보이나요? 작성자에게 조사해 달라고 요청하세요.
  - `main`이 현재 초록색이더라도, 풀 리퀘스트의 커밋이 `main`이 깨져 있던 시점의 커밋을 기반으로 할 수 있다는 가능성을 고려하세요. 이 경우라고 생각된다면, 작성자에게 그들이 작업을 시작한 후 적용된 수정 사항을 가져오기 위해 변경 사항을 `main` 위에 리베이스해 달라고 요청하세요.
- `main`이 깨진 것으로 보인다면, ["CI Test Failure"로 레이블이 지정된 이슈](https://github.com/facebook/react-native/issues?utf8=%E2%9C%93&q=is%3Aissue+is%3Aopen+label%3A%22%E2%9D%8CCI+Test+Failure%22+)를 찾아보세요.
  - `main`의 실패와 관련이 있어 보이는 이슈를 찾으면, 풀 리퀘스트로 돌아가 변경 사항을 제안해 주신 작성자에게 감사를 표하고, 테스트 실패가 그들의 특정 변경 사항과 무관할 수 있다고 알려주세요(작성자가 언제 테스트를 다시 실행해 볼 수 있는지 알 수 있도록 CI Test Failure 이슈에 다시 링크하는 것을 잊지 마세요).
  - `main`에서 관찰한 문제를 설명하는 기존 CI Test Failure 이슈를 찾을 수 없다면, 새 이슈를 제출하고 "CI Test Failure" 레이블을 사용하여 `main`이 깨졌음을 다른 사람들에게 알려주세요(예시는 [이 이슈](https://github.com/facebook/react-native/issues/23108)를 참고하세요).

## PR 우선순위 결정 방법

Meta의 React Native 팀 구성원은 풀 리퀘스트를 빠르게 검토하는 것을 목표로 하며 대부분의 PR은 일주일 이내에 응답을 받습니다.

## PR은 어떻게 병합되나요?

React Native GitHub 저장소는 실제로 Meta의 모노레포 중 하나의 하위 디렉터리를 미러링한 것입니다. 따라서 풀 리퀘스트는 전통적인 의미에서 병합되지 않습니다. 대신, Meta의 내부 코드 검토 시스템에 ["diff"](https://www.phacility.com/phabricator/differential/)로 임포트되어야 합니다.

임포트되면 변경 사항은 일련의 테스트를 거칩니다. 이 테스트 중 일부는 diff의 내용이 병합되기 전에 성공해야 하는 land-blocking입니다. Meta는 항상 `main`에서 React Native를 실행하며, 일부 변경 사항은 병합되기 전에 Facebook 직원이 풀 리퀘스트에 내부 변경 사항을 첨부해야 할 수 있습니다. 예를 들어, 모듈 이름을 변경하는 경우 모든 Facebook 내부 호출 사이트를 동일한 변경 사항에서 업데이트해야 합니다. diff가 성공적으로 적용되면 변경 사항은 결국 [ShipIt](https://github.com/facebook/fbshipit)에 의해 단일 커밋으로 GitHub에 다시 동기화됩니다.

Meta 직원은 두 가지 방법 중 하나로 풀 리퀘스트를 임포트할 수 있는 GitHub용 사용자 정의 브라우저 확장을 사용합니다: 풀 리퀘스트를 "fbsource에 적용"할 수 있는데, 이는 임포트되고 결과 diff가 자동으로 승인되며, 실패가 없으면 변경 사항이 결국 `main`으로 다시 동기화됩니다. 풀 리퀘스트는 "Phabricator에 임포트"될 수도 있는데, 이는 변경 사항이 적용되기 전에 추가 검토와 승인이 필요한 내부 diff로 복사됨을 의미합니다.

## 봇

풀 리퀘스트를 검토하고 작업하다 보면, 몇 가지 GitHub 봇 계정이 남긴 댓글을 접하게 될 수 있습니다. 이 봇들은 풀 리퀘스트 검토 프로세스를 지원하기 위해 설정되었습니다. 자세한 내용은 [봇 레퍼런스](/contributing/bots-reference)를 참고하세요.

## 풀 리퀘스트 레이블

- `Merged`: 닫힌 PR에 적용되어 변경 사항이 핵심 저장소에 통합되었음을 나타냅니다. 이 레이블은 풀 리퀘스트가 GitHub에서 직접 병합되지 않기 때문에 필요합니다. 대신, PR의 변경 사항이 담긴 패치가 임포트되어 코드 검토를 위한 대기열에 추가됩니다. 승인되면 Meta의 내부 모노레포 위에 해당 변경 사항을 적용한 결과가 새 커밋으로 GitHub에 동기화됩니다. GitHub은 해당 커밋을 원래 PR로 귀속시키지 않으므로, PR의 실제 상태를 전달하는 레이블이 필요합니다.
- `Blocked on FB`: PR이 임포트되었지만 변경 사항이 아직 적용되지 않은 상태입니다.
